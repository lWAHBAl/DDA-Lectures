Part 1 — Foundations of Distributed Systems
DEFINITION pp. 2–5

What Is a Distributed System?

A distributed system is a collection of independent computers that appears to its users as a single coherent system. Processes communicate only by message passing — there is no shared memory. Key characteristics:

  • Concurrency: programs run on different machines simultaneously
  • Independent failures: components can fail without taking down the whole system
  • No global clock: each machine has its own clock — they can drift apart
  • Lamport's definition: "You know you have a distributed system when the crash of a computer you've never heard of stops you from getting any work done."
PROPERTIES pp. 7–12

Core Properties

Three fundamental goals every distributed system aims for:

  • Transparency: Hide the physical distribution from users. Types include access, location, migration, replication, concurrency, failure, and persistence transparency.
  • Scalability: The system must remain effective as users and resources grow. Challenges: avoiding bottlenecks, controlling costs, preventing resource exhaustion.
  • Openness / Flexibility: Well-defined interfaces allow components from different developers to work together.
SOFTWARE LAYERS pp. 13–18

System Software Concepts

Three types of distributed system software sit between applications and hardware:

  • DOS (Distributed OS): Tightly-coupled OS for multi-processors. Tries to maintain a global view of all resources.
  • NOS (Network OS): Loosely-coupled. Each machine has its own OS with local services (e.g. Sun's NFS). Machines have a high degree of autonomy.
  • Middleware: An additional software layer on top of NOS that masks heterogeneity (different languages, hardware, networks). Provides distribution transparency. Examples: CORBA, Java RMI.
ARCHITECTURE pp. 21–27

Architectural Styles

Two dominant models for organizing roles in a distributed system:

  • Client–Server: Server provides a service and waits for requests. Client invokes service and blocks until reply. Advantage: efficient division of labor, easy synchronization. Problem: server is a bottleneck and single point of failure.
  • Peer-to-Peer (P2P): All processes are equal — each acts as both client and server. Subdivided into: structured P2P (organized data structure), unstructured P2P (random neighbors), and hybrid P2P (some special nodes). Much better scalability.
  • Proxy/Cache servers: A shared cache of recently used objects, placed close to clients. Reduces load on origin servers. A form of partial replication.
COMMUNICATION pp. 29–32

Message Passing

All communication in distributed systems is fundamentally based on low-level message passing — there is no shared memory between machines. Two operations: send and receive.

  • Synchronous: Both send AND receive are blocking. Sender waits until receiver receives. Tight coupling.
  • Asynchronous: Send is non-blocking — sender continues immediately while message is transmitted. Receive is usually blocking (to keep synchronization simple). Modern systems favor this with multi-threading so only one thread blocks.
Part 2 — Time & Clock Synchronization
WHY IT MATTERS pp. 34–43

The Problem of Time in DS

Physical clocks on different machines drift apart over time due to oscillator imperfections. This creates real problems:

  • Two servers log the same event with different timestamps — which came first?
  • UNIX make bug: a client clock lagging behind the server causes the build system to think files are already up-to-date when they aren't.
  • The clock drift rate for quartz crystals is ~10⁻⁶ seconds/second. Modern timer chips can drift ~0.86 seconds per day.
  • Key insight: It is impossible to guarantee physical clocks run at the same frequency, so we need algorithms to either synchronize them or work around the problem.
PHYSICAL CLOCKS pp. 45–52

Physical Time Standards

  • TAI (International Atomic Time): Based on counting transitions of cesium-133 atoms. 9,192,631,770 transitions = 1 second. Extremely stable.
  • UTC (Universal Coordinated Time): TAI adjusted with leap seconds to stay in phase with solar time. Broadcast by satellites and radio signals.
  • Clock Skew: The difference in readings between two clocks at any given moment.
  • Clock Drift: Different rates of counting — clocks run at slightly different speeds. Small differences accumulate into large observable skew.
ALGORITHM pp. 53–59

Cristian's Algorithm (External Sync)

A process P periodically contacts a time server S that has access to UTC:

  • P sends a request to S at time T₁
  • S replies with its current time C(S) at time T₂
  • P receives reply at time T₃
  • P estimates: C(P) = C(S) + (T₃ − T₁)/2 (adds half round-trip time)
  • Accuracy = Tround/2 − min (where min = minimum one-way delay)
  • Limitations: single point of failure, variability in network delays. Solution: take minimum of several requests.
ALGORITHM pp. 55–57

NTP — Network Time Protocol

A hierarchical tree of time servers synchronizes all machines on the internet:

  • Stratum 1: Primary servers directly connected to UTC source
  • Stratum 2: Synchronized from Stratum 1
  • Stratum 3: Synchronized from Stratum 2, and so on
  • NTP runs multiple trials and selects the trial with the minimum round-trip delay to get the best estimate
  • Offset θ = (a+b)/2, Delay δ = a−b, where a = T1−T3, b = T2−T4
ALGORITHM pp. 60–61

Berkeley Algorithm (Internal Sync)

Used when no UTC source is available — focuses on keeping all clocks consistent with each other rather than accurate to real time:

  • An elected master (time daemon) polls all other machines for their current clock values
  • The daemon adjusts the received times to account for round-trip latency
  • The daemon computes the average of all times
  • Each machine is told how much to adjust its clock (not what time it is)
  • Clocks never go backwards — if a clock is ahead, the daemon slows it down instead of jumping it back
  • Average time is more accurate than any single clock, though it still drifts from UTC
Part 3 — Logical Clocks & Event Ordering
CONCEPT pp. 62–66

Happens-Before Relation (→)

Lamport observed that for many DS algorithms, we don't need to know the absolute real time of events — we just need to know their relative ordering. The "happens-before" relation (→) captures causal ordering:

  • Rule 1: On the same process: a→b if a occurred before b
  • Rule 2: If process P sends message m, and Q receives m: send(m) → receive(m)
  • Rule 3 (transitivity): If a→b and b→c, then a→c
  • If neither a→b nor b→a, events are concurrent — they cannot causally affect each other
ALGORITHM pp. 66–69

Lamport Timestamps

Each process maintains a counter (logical clock) that provides a partial ordering of all events:

  • Each process starts with counter = 0
  • Rule 1: Before any event, increment the counter: L := L + 1
  • Rule 2a: When sending a message, piggyback the current counter value t
  • Rule 2b: On receiving message with timestamp t: L := max(L, t) + 1
  • Guarantee: if a→b then L(a) < L(b)
  • Limitation: L(a) < L(b) does NOT imply a→b (concurrent events can have ordered timestamps)
ALGORITHM pp. 70–87

Vector Clocks

Fix the limitation of Lamport clocks — vector clocks can detect concurrency. Each process maintains a vector of N counters (one per process):

  • V[i][i] incremented before each event at process i
  • Messages carry the full vector as their timestamp
  • On receiving message with vector t: V[i][j] = max(V[i][j], t[j]) for all j, then increment V[i][i]
  • V[i][j] means: "the number of events at process j that causally precede what I know"
  • e → e' if and only if V(e) < V(e') (component-wise less-than-or-equal, with at least one strict)
  • If neither V(e) ≤ V(e') nor V(e') ≤ V(e), the events are truly concurrent
Part 4 — Replication
MOTIVATION pp. 93–96

Why Replicate Data?

Replication = maintaining copies of data at multiple sites. Three main reasons:

  • Performance Enhancement: Load balancing — divide work among servers. Exploit geographic proximity (closer server = faster access). Caching via proxy servers.
  • Availability: If one replica fails, others serve requests. If each of n servers has failure probability p, availability = 1 − pⁿ (much better than a single server's 1−p).
  • Fault Tolerance: Stronger than availability — guarantees strictly correct behavior. Example: tolerate up to f crash failures with f+1 replicas. Byzantine fault tolerance requires 2f+1 replicas.
MODEL pp. 99–105

Replication Architecture

How a replicated system is structured:

  • Replica Managers (RM): Each RM holds copies of the replicated objects and applies operations to them directly and recoverably
  • Front Ends (FE): Clients interact with front ends, which provide replication transparency — the client doesn't know there are multiple copies
  • One-Copy Serializability: The effect of transactions on replicated objects must be the same as if performed on a single copy
  • Five phases per request: Request → Coordination → Execution → Agreement → Response
  • Request ordering options: FIFO, Causal, or Total ordering
Part 5 — Consistency Models
NOTATION p. 108

Reading the Diagrams

Consistency model diagrams use this notation:

  • W(x)a = write value 'a' to variable x
  • R(x)a = read value 'a' from variable x
  • Time progresses left to right. Each row is one process.
  • The subscript (W₁, R₂) indicates which process performed the operation
MODEL pp. 109–110

Strict Consistency

The strongest model: any read on x returns the value of the most recent write on x, globally. Requires absolute global time ordering. Impossible to implement in practice — a write on host B cannot be instantaneously visible to host A over a physical network (even light takes time to travel). If A reads x at t₁ and B writes x at t₂ = t₁ + 3ns, A must get the old value — but this requires infinitely fast communication.

MODEL p. 111

Linearizability

A weaker version of strict consistency. A service is linearizable if there exists some interleaving of all operations that (1) meets the spec of a single correct copy and (2) is consistent with the real time at which each operation occurred. The real-time requirement means clients must receive up-to-date information. Practically difficult due to clock synchronization challenges.

MODEL pp. 112–117

Sequential Consistency

All processes see all operations in the same order, and that order is consistent with each individual process's program order. Does NOT require real-time ordering — the global sequence just needs to be consistent with local order. Example: if P1 does x=1 then P2 does y=1, all processes must agree on the relative order these are seen — but the agreed-upon order doesn't need to match wall-clock time. All valid interleavings are acceptable as long as everyone agrees on which one occurred.

MODEL pp. 118–121

Causal Consistency

Only causally related writes must be seen in the same order by all processes. Concurrent writes (writes with no causal relationship) may be seen in different orders by different processes. Example: if P1 writes x=a and P2 reads x=a and then writes x=b, then P2's write is causally related to P1's write — all processes must see W(x)a before W(x)b. But if P2's write had occurred without reading P1's value first, the two writes are concurrent and can be seen in any order.

MODEL pp. 122–125

FIFO Consistency

Writes from a single process are seen by all other processes in the order they were issued. Writes from different processes may be seen in different orders by different processes. This is the weakest ordering guarantee without synchronization. Danger: if P1 does "x=1; if (y==0) kill(P2)" and P2 does "y=1; if (x==0) kill(P1)", under FIFO both processes could be killed because each may read the other's variable before seeing the write (unlike sequential consistency which would prevent this).

MODEL pp. 126–127

Weak Consistency

Shared data can only be relied upon to be consistent after a synchronization operation. Between synchronizations, processes can see stale data. Synchronization variables are sequentially consistent. No read/write on data items is allowed until all previous synchronization operations complete. A synchronization "flushes the pipeline" — after an update, a process can force all other processes to see it by performing a synchronize operation.

MODEL pp. 128–131

Release Consistency

Refinement of weak consistency — distinguishes between entering and exiting a critical section using acquire and release operations:

  • Before reading/writing shared data: all previous acquires must have completed
  • Before a release: all previous reads/writes must have completed
  • Eager implementation: After release, updates are immediately propagated to all processes
  • Lazy implementation: Nothing sent on release; on acquire, the process fetches only the data it needs using timestamps
MODEL pp. 132–134

Entry Consistency

The most fine-grained model — each shared data item is associated with its own synchronization variable (lock). Only data guarded by a specific lock is made consistent when that lock is acquired. This allows multiple critical sections with disjoint data to execute concurrently. On acquire: all remote changes to the guarded data are made visible. Each lock has a "current owner" who can repeatedly enter/exit without network messages unless another process competes.

Summary: Consistency Models Ordered by Strength
FULL COMPARISON p. 135

Consistency Models — Weakest to Strongest

Models without synchronization operations (stricter):

  • Strict: Absolute real-time ordering of all accesses. Impossible in practice.
  • Linearizability: All accesses ordered by global timestamp. Real-time consistent.
  • Sequential: All processes agree on the same ordering, consistent with program order.
  • Causal: Only causally-related writes seen in same order. Concurrent writes may differ.
  • FIFO: Writes from same process seen in order. Writes from different processes may differ.

Models with synchronization operations (weaker, but more practical):

  • Weak: Data consistent only after explicit synchronization.
  • Release: Data consistent when a critical region is exited (release).
  • Entry: Data consistent when a critical region is entered (acquire).
