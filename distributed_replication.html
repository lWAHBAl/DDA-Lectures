Part 1 — Why Replication?
MOTIVATION Slides 2–5

Why Replicate Data?

Replication is the maintenance of copies of data at multiple sites. It serves three key goals:

  • Performance Enhancement: Divide work across multiple servers (load balancing); exploit geographic proximity to clients; enable caching via proxy servers or DNS.
  • Fault Tolerance: If one server fails, others can continue serving requests without interruption.
  • Increased Availability: With n servers each having independent failure probability p, availability = 1 − pⁿ. Note: caching alone does NOT count as improving availability.
REPLICATION VS. PARTITIONING Slide 5

Distributing Data: Two Strategies

Two complementary approaches to spreading data across nodes:

  • Replication: Store copies of the same data on several nodes. Introduces redundancy; improves scalability (parallel I/O), availability (full failover), and latency (serve from nearest node). No memory scalability benefit.
  • Partitioning (Sharding): Split the dataset into subsets stored on different nodes. Improves both I/O scalability AND memory consumption. Node failures only take out part of the data.

This lecture focuses on replication strategies.

SERVER PLACEMENT Slides 7–8

Where to Place Replica Servers?

Given N candidate locations, how do you choose K server locations? Use the greedy k-median heuristic:

  • Compute the total access cost from each site to all others.
  • Place the first server at the site with minimum total cost.
  • Repeat: recalculate costs accounting for existing replicas (each client accesses its nearest replica), place next server at minimum cost.
  • Continue until K servers are placed.
Availability = 1 − p^n
(n servers, each with failure prob. p)
Part 2 — Replication Architectures
SINGLE-LEADER Slides 11–23

Single-Leader Replication

One designated leader (also called master or primary) controls all writes. Followers (slaves, secondaries, hot standbys) handle reads.

  • Write path: Client → Leader → updates local storage → sends change stream to followers.
  • Read path: Client → any replica (leader or follower).
  • Replication lag: Delay between a write on the leader and the same write on a follower. Usually under 1 second, but can grow under load or faults.
  • Read inconsistency: A read to a lagging follower may return stale data — application logic must handle this.
WRITE PROPAGATION Slides 17–18

Synchronous vs. Asynchronous Writes

Three strategies for propagating writes to followers, each trading consistency for availability:

  • Synchronous: Write returns only when ALL replicas confirm. Guarantees full consistency. Drops availability — one slow/crashed follower blocks every write.
  • Semi-synchronous: Write returns when ONE replica confirms. At least one follower can restore state if leader fails. Balanced trade-off.
  • Asynchronous: Write returns immediately, no acknowledgment waited for. Maximum availability. Drops consistency — writes can be lost if leader crashes before propagation.
FAILOVER Slide 21

Handling Leader Failure

When the leader fails, a failover process must elect a new leader:

  • Detect failure: If the leader doesn't respond within a timeout, assume it dead.
  • Elect new leader: Followers elect the one with the most up-to-date data, or a controller node decides.
  • Reconfigure: Redirect writes, make old leader a follower on recovery.
  • Split Brain: If two replicas both believe they are leader, data can be corrupted.
  • Timeout dilemma: Short timeouts cause unnecessary failovers during load spikes. Many teams perform failovers manually.
REPLICATION LOGS Slides 22–23

How Are Changes Propagated?

Three implementation approaches for replication logs:

  • Statement-based: Leader sends INSERT/UPDATE/DELETE statements. Problem: non-deterministic functions (NOW(), RAND()), auto-increments and triggers may differ per replica.
  • WAL (Write-Ahead Log) shipping: Leader sends physical disk block changes. Problem: tightly coupled to storage engine version — all nodes must run the same software.
  • Logical (row-based) log: Leader sends logical row changes (old value, new value, transaction ID). Decoupled from storage format. Extra memory when used alongside a physical WAL.
MULTI-LEADER Slides 25–26

Multi-Leader Replication

Multiple leaders each accept writes and propagate to their own followers and to each other. Useful for multiple datacenters, offline clients, or improved write availability.

  • Write conflicts: Two leaders can modify the same item concurrently in different ways. Conflicts are detected by comparing old values in the change log.
  • Last write wins: Accept the write with the highest ID/timestamp.
  • Merge: Combine both values (e.g., alphabetically concatenated).
  • Application-managed: Store a conflict structure and let the application resolve it on next read.

Conflict resolution is always asynchronous — both writes already succeeded, so conflicting writes cannot be rejected.

LEADERLESS Slides 28–32

Leaderless Replication

No leader/follower distinction. All replicas accept reads and writes. Used in Dynamo-style databases (Cassandra, Riak).

  • Writes and reads are sent to ALL replicas in parallel.
  • A query succeeds when a quorum (minimum number) of sub-queries succeed — tolerates some failing replicas.
  • On read, the client receives multiple version-stamped values and selects the newest version.
  • Pitfalls: Clock skew can cause "last write wins" to overwrite newer values; concurrent writes are inherently tricky; a seemingly failed write may have silently succeeded on some nodes.
Part 3 — Consistency Models
CAP & CONSISTENCY Slides 16, 38–40

Consistency Basics & CAP

With replication, network partitions are unavoidable — so the CAP theorem forces a choice: drop Availability or drop Consistency?

  • Replication transparency: Clients must not know that copies exist — they experience a single logical object.
  • One-copy serializability: Replicated transactions must behave as if executed one at a time on a single copy.
  • CP systems: BigTable, HBase, MongoDB. Consistent but may be unavailable under partition.
  • AP systems: Cassandra, CouchDB, Riak. Always available, but may serve stale data.
STRICT & LINEAR Slides 44–46

Strict & Linearizable Consistency

The strongest guarantees — rarely achievable in practice:

  • Strict Consistency: Every read returns the value of the most recent write, globally. Assumes instantaneous visibility — physically impossible in any distributed system (even light travel time violates it).
  • Linearizability: Weaker but practical form: each operation appears to take effect instantaneously at some point between its invocation and response, consistent with real time. Clients always see up-to-date values, but requires synchronized clocks.
Strict: R(x) always returns latest W(x)
Linear: consistent with real-time order
SEQUENTIAL Slides 47–52

Sequential Consistency

Weaker than linearizability — does not require real-time ordering, only per-client program order.

  • All processes see the same interleaving of all operations — but the interleaving need not match real clock time.
  • Each client's operations appear in their program order within the global interleaving.
  • A store is NOT sequentially consistent if different replicas show writes in different orders to different clients simultaneously.

Example: P3 reads b then a; P4 reads b then a — sequentially consistent. But if P4 reads a then b while P3 reads b then a — NOT sequentially consistent.

CAUSAL Slides 53–56

Causal Consistency

Weaker than sequential — only causally related writes must be seen in the same order by all processes. Concurrent writes may be seen in any order.

  • If P1 writes x=a and P2 reads x=a then writes x=b, these writes are causally related — all processes must see a before b.
  • If P1 writes x=a and P2 writes x=b without reading x first, those writes are concurrent — different processes may see them in any order.
  • Causal consistency is achievable without sacrificing performance, unlike sequential or linearizability.
FIFO Slides 57–60

FIFO (PRAM) Consistency

The weakest useful model — writes from a single process are seen in their issue order by all others, but writes from different processes may arrive in any order at different replicas.

  • Also called Pipelined RAM (PRAM) — a process does not block waiting for a write to complete.
  • Tags each write with (process, sequence#) to enforce ordering per sender.
  • A valid FIFO execution can look non-sequential: P3 sees W2 before W1 while P4 sees W1 before W2 — fine as long as P2's own writes arrive in order at both.
  • Danger: with FIFO alone, two racing processes can both be killed in a mutual exclusion scenario — sequential consistency would prevent this.
CONSISTENCY HIERARCHY All

Consistency Model Hierarchy

From strongest to weakest — each relaxation trades correctness guarantees for performance and availability:

  • Strict: Absolute real-time ordering. Impossible to implement.
  • Linearizable: Real-time consistent. Requires synchronized clocks.
  • Sequential: Program-order consistent for all clients. Widely used in practice.
  • Causal: Only causally-related operations ordered. Concurrent writes may differ.
  • FIFO: Per-sender ordering only. Highly scalable.
Strict ⊂ Linear ⊂ Sequential ⊂ Causal ⊂ FIFO
