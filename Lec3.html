Part 1 — The Problem of Time in Distributed Systems
MOTIVATION Slides 3–5

Why Does Synchronization Matter?

In a distributed system, each machine runs its own independent clock. These clocks drift relative to each other, creating severe problems:

  • Event ordering: Which sharpshooter in a multiplayer game killed the target first?
  • Log consistency: Server A timestamps a purchase at 9:15:32 but Server B's "flight full" log shows 9:10:10 — causing logical impossibilities.
  • Mutual exclusion: Many distributed algorithms require agreement on event order.
  • Fundamental limit: We cannot timestamp events across nodes precisely enough to always determine their true causal order.
PHYSICAL TIME Slides 8–12

Physical Clocks & UTC

How real time is defined and maintained:

  • Solar day: 1/86400 of the mean solar day — but Earth's rotation varies.
  • TAI (International Atomic Time): Defined by cesium-133 oscillations. 9,192,631,770 cycles = 1 second. Extremely stable.
  • UTC: TAI ± leap seconds (to stay aligned with the sun). Broadcast by radio stations like WWV in Colorado (±10 ms accuracy over the air).
  • Computer clocks: Modern timer chips drift up to 0.86 seconds per day (relative error ~1/100,000).
  • Clock skew: dC/dt > 1 = fast clock; dC/dt < 1 = slow clock; perfect clock: dC/dt = 1.
INTERACTION MODEL Slide 7

Distributed System Model

The formal model underlying all synchronization algorithms:

  • N processes p₁, p₂, …, pN — each runs on its own processor.
  • No shared memory — processes communicate only by sending messages.
  • State sᵢ: the local variable values and OS environment of process pᵢ.
  • Actions: either state-transforming (modifying local variables) or communication actions (send/receive).
  • Goal: coordinate these isolated processes to agree on time and event order.
Part 2 — Physical Clock Synchronization Algorithms
EXTERNAL SYNC Slides 13–22

Cristian's Algorithm

A process P synchronizes with an external UTC time server S:

  • P sends a request to S at time T₀.
  • S replies with its clock value C(S). P receives the reply at time T₁.
  • Round-trip time: RTT = T₁ − T₀.
  • P cannot simply set its clock to C(S) — the reply took time to arrive.
  • Estimate: C(P) := C(S) + RTT/2
  • Accuracy: ±(RTT/2 − min) where min is the minimum one-way latency.
estimate = t + T_round / 2
accuracy = ±(T_round/2 − min)

Limitations: single point of failure, variable network delays. Mitigated by repeated requests and taking the minimum RTT.

NTP Slides 15–19

Network Time Protocol (NTP)

The most widely-used clock synchronization protocol on the Internet (10–20 million servers/clients):

  • Uses a hierarchy of time servers (strata). Stratum 1 = direct UTC source; Stratum 2 = synced from Stratum 1, etc.
  • NTP computes clock offset θ and round-trip delay δ using four timestamps: T₁ (client send), T₂ (server receive), T₃ (server send), T₄ (client receive).
  • Let a = T₁ − T₃, b = T₂ − T₄.
θ = (a + b) / 2
δ = a − b
  • Accuracy: ~milliseconds on WANs, submilliseconds on LANs, submicroseconds with precision time sources.
  • Performs multiple trials and selects the result with minimum delay to reduce variability.
INTERNAL SYNC Slides 24–26

Berkeley Algorithm

An internal synchronization method — no UTC reference needed. All machines agree on a common time, even if it's not UTC-accurate:

  • An elected master/time daemon periodically polls all slave machines for their clock values.
  • Master adjusts received times for RTT/latency and computes an average (ignoring outliers).
  • Master sends each slave the adjustment amount (positive or negative), not the absolute time — avoiding further delay-induced error.
  • Faulty clocks are excluded if they differ by more than a threshold.
  • If the master fails, a new one is elected (though election is not bounded in time).

Example: daemon reads 3:00, slaves report 2:50 and 3:25 → average ~3:05 → adjustments: +5, +15, −20.

Part 3 — Logical Clocks & Event Ordering
LAMPORT 1978 Slides 27–38

Lamport's Logical Clocks

Since physical clocks can't be perfectly synchronized, Lamport proposed ordering events by causality, not wall-clock time.

  • Happens-Before (→): If event a occurs before b on the same process, a→b. If p₁ sends m to p₂, send(m)→receive(m). Transitivity: a→b and b→c implies a→c.
  • Concurrent events: if neither a→b nor b→a, then a and b are concurrent (written a ∥ b).
Rules:
① Lᵢ := Lᵢ + 1 before each event
② Piggyback Lᵢ on sent message
③ On receive: Lⱼ := max(Lⱼ, t) + 1

Guarantee: if e → e′ then L(e) < L(e′). But the converse is NOT guaranteed — equal or higher timestamps don't imply causality.

MATTERN & FIDGE 1989/91 Slides 39–57

Vector Clocks

Lamport clocks give a partial ordering, but can't detect concurrent events. Vector clocks solve this:

  • Each process pᵢ maintains a vector Vᵢ of N integers (one per process), initially all zero.
  • Before event: Vᵢ[i] := Vᵢ[i] + 1
  • Send: piggyback entire vector Vᵢ as timestamp t.
  • Receive (m, t): Vᵢ[j] := max(Vᵢ[j], t[j]) for all j.
  • Vᵢ[j] (j ≠ i) = #events at Pⱼ that causally precede this event at Pᵢ.
e → e′ ⟺ V(e) < V(e′)
e ∥ e′ iff V(e) < V(e′) AND V(e′) < V(e) — both false

Vector clocks provide a complete characterization of the happens-before relation.

APPLICATION Slides 41–53

Totally Ordered Multicast

Using Lamport clocks to deliver messages in the same causal order to all replicas:

  • Each sent message is timestamped with the sender's logical clock value.
  • Received messages are placed in a local queue ordered by timestamp.
  • The receiver multicasts an acknowledgment to all other processes.
  • A message is committed (delivered to the application) only when it is at the head of the queue AND acknowledged by every other process.
  • This guarantees all replicas see the same ordering — enabling consistent distributed writes.

Key insight: Lamport acks always carry higher timestamps than the original message — ensuring safety.
